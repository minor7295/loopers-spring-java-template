# 멘토 질문 리스트: Transactional Operations (대량 트래픽 관점)

## 📌 개요

본 문서는 대량 트래픽 상황에서 transactional operation 설계 시 멘토에게 질문할만한 내용들을 정리한 문서입니다.
단순히 찾아보면 답이 나오는 기술적 질문이 아니라, **가치 평가, 비즈니스 모델, 유스케이스 차이**에 따라 달라질 수 있는 판단이 필요한 질문들에 초점을 맞췄습니다.

---

## 🎯 카테고리 1: 비즈니스 모델에 따른 정합성 요구사항

### Q1-1. "약간의 오차 허용"과 "절대 오차 불가"의 경계는 어떻게 판단하나요?

**상황**:
- 좋아요 수는 약간의 지연/오차 허용 가능 → Eventually Consistent
- 재고/포인트는 절대 오차 불가 → Strong Consistency

**질문**:
- 실제 서비스에서 "약간의 오차"가 허용되는 기준은 무엇인가요?
- 예를 들어, 좋아요 수가 실제보다 1~2개 적게 표시되어도 비즈니스적으로 문제가 없는지, 아니면 특정 임계값(예: 5% 오차)까지 허용하는지?
- 금전적 손실이 없는 도메인이라도 정합성이 중요한 경우는 어떤 케이스인가요?

**고려사항**:
- 사용자 경험에 미치는 영향
- 법적/규제 요구사항
- 고객 신뢰도
- 운영 복잡도

### Q1-2. Hot Spot 발생 시 "정확성 포기" vs "성능 포기"의 판단 기준은?

**상황 설명**:

현재 프로젝트에서는 재고 차감과 포인트 차감에 비관적 락(Pessimistic Lock)을 사용하고 있습니다. 이는 데이터 정합성을 완벽하게 보장하지만, 인기 상품에 주문이 몰리는 Hot Spot 상황에서는 문제가 발생할 수 있습니다.

예를 들어, 특정 상품에 100명이 동시에 주문을 요청하면, 비관적 락으로 인해 모든 요청이 순차적으로 처리되어야 합니다. 첫 번째 주문이 락을 획득하고 재고를 차감한 후 커밋할 때까지 나머지 99명은 대기해야 하므로, DB 병목이 발생하고 전체 처리량이 급격히 감소합니다. API 서버를 10대, 100대로 늘려도 결국 같은 DB row에 대한 락 경쟁이 발생하기 때문에 처리량이 증가하지 않는 문제가 있습니다.

이런 상황에서 Optimistic Lock으로 전환하면 락 경쟁 없이 병렬 처리가 가능해 성능이 크게 향상됩니다. 하지만 동시에 여러 요청이 같은 재고를 읽고 차감을 시도하다 보면, 일부 요청은 OptimisticLockingFailureException이 발생하여 실패할 수 있습니다. 이 경우 재시도 로직을 통해 처리할 수 있지만, 재고가 부족한 경우에는 최종적으로 실패할 수도 있습니다.

**핵심 질문**:

1. **Hot Spot 발생 시 판단 기준**: 실제 서비스에서 Hot Spot이 발생했을 때, "비관적 락을 유지할 것인가" 아니면 "낙관적 락으로 전환할 것인가"를 어떤 기준으로 판단하시나요? 

   구체적으로, 트래픽이 초당 몇 건 이상이면 전환을 고려하시는지, 아니면 다른 지표(예: 락 대기 시간, DB CPU 사용률, 타임아웃 발생률 등)를 기준으로 하시는지 궁금합니다. 또한 실패율이 몇 퍼센트 이하일 때 허용 가능하다고 판단하시는지, 그리고 사용자 경험에 미치는 영향(예: 주문 실패 시 재시도 화면 표시)을 어떻게 평가하시는지 알고 싶습니다.

2. **"재시도 가능한 실패"와 "절대 실패 불가"의 경계**: 재고 차감 실패가 발생했을 때, 사용자가 재시도 버튼을 눌러 다시 시도할 수 있다면 "재시도 가능한 실패"로 볼 수 있습니다. 하지만 이런 실패가 너무 자주 발생하거나, 재고가 실제로는 있는데도 실패하는 경우가 많다면 사용자 경험이 크게 저하될 수 있습니다. 

   반대로, 비관적 락을 사용하면 정확성은 보장되지만 모든 요청이 순차 처리되어 대기 시간이 길어지고, 일부 요청은 타임아웃으로 실패할 수 있습니다. 이 경우에도 "절대 실패 불가"라고 볼 수 있을까요? 실제 서비스에서는 어떤 수준의 실패를 허용 가능하다고 판단하시는지, 그리고 "재시도 가능한 실패"와 "절대 실패 불가"의 경계를 어떻게 설정하시는지 궁금합니다.

3. **비즈니스 모델별 차이**: 이커머스 서비스에서는 재고 oversell이 발생하면 환불 처리 비용과 고객 신뢰도 하락이라는 간접적 손실이 발생합니다. 하지만 금융 서비스에서는 포인트나 잔액 차감 시 오차가 발생하면 법적 책임까지 갈 수 있습니다. 

   실제로 서비스 특성(이커머스, 금융, 소셜, 게임 등)에 따라 "정확성 포기" vs "성능 포기"의 판단 기준이 달라지시나요? 예를 들어, 이커머스에서는 약간의 재고 oversell을 허용하고 나중에 보상하는 방식도 가능하지만, 금융 서비스에서는 그런 방식이 불가능할 수 있습니다. 실제 경험하신 사례나 일반적인 패턴이 있다면 공유해주시면 감사하겠습니다.

**추가로 궁금한 점**:

- Hot Spot이 예측 가능한 경우(예: 프로모션, 이벤트)와 예측 불가능한 경우(예: 바이럴 상품)에 대응 전략이 달라지나요?
- Optimistic Lock으로 전환했을 때, 재시도 로직의 최대 재시도 횟수나 재시도 간격은 어떻게 결정하시나요?
- Hot Spot 발생 시 임시로 전략을 변경(예: Pessimistic → Optimistic)하는 것이 가능한가요, 아니면 사전에 설계해야 하나요?

**고려사항**:
- 서비스 특성 (이커머스 vs 금융 vs 소셜)
- 고객 기대치와 경쟁 서비스 수준
- 운영팀의 모니터링 및 대응 역량
- 비즈니스 임팩트 (매출 손실, 고객 이탈 등)

### Q1-3. "금전적 손실"의 정의와 범위는 어디까지인가요?

**상황**:
- 포인트 중복 차감 → 명백한 금전적 손실
- 좋아요 중복 → 비즈니스 데이터 오염 (간접적 손실?)
- 재고 oversell → 환불 처리 비용 (간접적 손실?)

**질문**:
- "금전적 손실"의 정의가 서비스마다 다른가요?
- 간접적 손실(환불 처리 비용, 고객 신뢰도 하락)도 "금전적 손실"로 간주하나요?
- 손실 규모가 작으면(예: 1원) 정합성 보장을 포기할 수도 있나요?
- 실제 서비스에서 "금전적 손실 방지"를 위해 어느 정도까지 성능을 포기하나요?

**고려사항**:
- 손실 규모
- 발생 빈도
- 복구 가능성
- 법적 책임

## 🎯 카테고리 2: 트래픽 패턴과 전략 선택

### Q2-1. 트래픽 패턴을 어떻게 예측하고 전략을 선택하나요?

**상황**:
- 현재는 트래픽이 낮아서 Pessimistic Lock 사용
- 향후 트래픽 증가 시 Optimistic Lock 또는 Queueing 전환 필요

**질문**:
- 실제 서비스에서 트래픽 패턴을 어떻게 예측하나요?
  - 과거 데이터 분석?
  - 비즈니스 팀과의 협업?
  - 시뮬레이션/부하 테스트?
- "트래픽이 낮다"와 "트래픽이 높다"의 기준은?
  - 절대적 기준? (예: 초당 100건)
  - 상대적 기준? (DB 처리 용량 대비)
- 전략 전환 시점을 어떻게 결정하나요?
  - 사전 예방적 전환?
  - 문제 발생 후 전환?

**고려사항**:
- 비즈니스 성장 속도
- 인프라 비용
- 전환 리스크
- 모니터링 역량

**상황**:

**질문**:

- Hot Spot 발생 시 즉시 대응 전략이 있나요?
  - 자동 전환? (Pessimistic → Optimistic)
  - 수동 개입?
- Hot Spot 예측이 가능한가요? (예: 프로모션, 이벤트)

**고려사항**:
- 모니터링 인프라
- 자동화 수준
- 운영팀 역량
- 비즈니스 이벤트 일정

### Q2-3. "약간의 지연 허용"의 기준은 무엇인가요?

**상황**:

**질문**:

  - 몇 초까지 허용? (1초? 5초? 30초?)
  - 사용자 경험에 미치는 영향은?
- 도메인별로 지연 허용 기준이 다른가요?
  - 좋아요 수: 5초 지연 OK?
  - 재고 수: 1초 지연도 불가?
  - 주문 상태: 실시간 필수?
- 사용자가 지연을 인지할 수 있는가가 기준인가요?

**고려사항**:
- 사용자 기대치
- 서비스 특성
- 경쟁 서비스 수준
- 기술적 제약

## 🎯 카테고리 4: 트랜잭션 범위와 성능 트레이드오프

### Q4-1. "트랜잭션 범위 최소화"와 "원자성 보장"의 균형점은?

**상황**:
- 트랜잭션을 넓게 잡으면 락 유지 시간 증가 → 데드락 위험
- 트랜잭션을 좁게 잡으면 원자성 보장 안 됨 → 데이터 불일치

**질문**:
- 실제 서비스에서 트랜잭션 범위를 어떻게 결정하나요?
  - 유즈케이스 단위? (현재 프로젝트 방식)
  - 더 작은 단위?
  - 더 큰 단위?
- "원자성이 보장되어야 하는 단위"는 어떻게 정의하나요?
  - 비즈니스 요구사항 기반?
  - 기술적 제약 기반?
- 트랜잭션 범위가 길어도 괜찮은 경우는?
  - 예: 외부 I/O 없음, 락 범위 최소화

**고려사항**:
- 비즈니스 요구사항
- 기술적 제약
- 운영 리스크
- 복잡도

---

### Q4-2. WAL 성능과 트랜잭션 범위의 실제 트레이드오프는?

**상황**:
- 트랜잭션을 잘게 나누면 커밋 I/O 증가 → WAL 성능 저하
- 하지만 트랜잭션을 넓게 잡으면 락 유지 시간 증가

**질문**:
- 실제 서비스에서 WAL 성능이 병목이 되는 경우가 있나요?
- 커밋 I/O가 몇 번 이상이면 문제가 되나요?
  - 예: 하나의 유즈케이스당 1번 vs 3번의 차이는?
- WAL 성능 저하 vs 락 유지 시간 증가 중 어느 것이 더 위험한가요?
- 대량 트래픽에서 실제로 어떤 쪽이 병목이 되나요?

**고려사항**:
- DB 성능 특성
- 트래픽 패턴
- 하드웨어 사양
- 모니터링 데이터

---

### Q4-3. "외부 I/O를 트랜잭션 밖으로"의 실무적 적용은?

**상황**:
- 이론적으로는 외부 API 호출을 트랜잭션 밖으로 빼야 함
- 하지만 실제로는 비즈니스 로직상 트랜잭션 내부에 있어야 할 수도 있음

**질문**:
- 실제 서비스에서 외부 I/O를 트랜잭션 밖으로 빼는 것이 항상 가능한가요?
- 불가능한 경우는 어떤 케이스인가요?
  - 예: 외부 결제 API 호출 후 주문 저장
- 외부 I/O가 트랜잭션 내부에 있어도 괜찮은 경우는?
  - 예: 매우 빠른 외부 API (몇 ms)
  - 예: 실패해도 롤백 가능한 경우
- 실무에서 어떤 패턴을 주로 사용하나요?
  - Saga 패턴?
  - Outbox 패턴?
  - 그냥 트랜잭션 내부에?

**고려사항**:
- 외부 API 특성
- 실패 처리
- 복잡도
- 운영 리스크

---

## 🎯 카테고리 5: 격리 수준과 일관성 보장

### Q5-1. REPEATABLE READ vs SERIALIZABLE의 실제 선택 기준은?

**상황**:
- 현재 프로젝트는 REPEATABLE READ 사용
- Phantom Read 가능하지만 성능 우선

**질문**:
- 실제 서비스에서 SERIALIZABLE을 사용하는 경우가 있나요?
- 어떤 도메인에서 SERIALIZABLE이 필수인가요?
  - 금융? 의료? 법률?
- REPEATABLE READ에서 발생하는 Phantom Read가 실제로 문제가 되는 경우는?
  - 예: 집계 쿼리 결과 불일치
- 성능 저하를 감수하고라도 SERIALIZABLE을 선택하는 기준은?

**고려사항**:
- 비즈니스 요구사항
- 법적/규제 요구사항
- 성능 영향
- 복잡도

---

### Q5-2. "약간의 불일치 허용"과 "절대 불일치 불가"의 경계는?

**상황**:
- 좋아요 목록과 집계 결과가 약간 다를 수 있음 (Phantom Read)
- 하지만 사용자 경험에 큰 영향 없을 수 있음

**질문**:
- 실제 서비스에서 "약간의 불일치"가 허용되는 기준은?
  - 사용자가 인지할 수 있는가?
  - 비즈니스 로직에 영향을 주는가?
- 도메인별로 불일치 허용 기준이 다른가요?
  - 좋아요 수: 1~2개 차이 OK?
  - 재고 수: 1개 차이도 불가?
  - 주문 금액: 0원 차이만 허용?
- 불일치 허용 시 발생할 수 있는 문제는?
  - 예: 사용자 문의 증가
  - 예: 비즈니스 로직 오류

**고려사항**:
- 사용자 경험
- 비즈니스 로직
- 법적 책임
- 운영 복잡도

---

## 🎯 카테고리 6: 모니터링과 알림 전략

### Q6-1. 대량 트래픽에서 모니터링해야 할 핵심 지표는?

**상황**:
- 대량 트래픽에서 성능 저하나 장애를 조기에 감지해야 함
- 어떤 지표를 모니터링할지 결정 필요

**질문**:
- 실제 서비스에서 어떤 지표를 모니터링하나요?
  - 트랜잭션 커밋 횟수?
  - 락 대기 시간?
  - 데드락 발생 횟수?
  - WAL I/O 대기 시간?
- 알림 임계값은 어떻게 설정하나요?
  - 예: 락 대기 시간 1초 이상?
  - 예: 데드락 1회 발생?
- 모니터링 지표가 너무 많으면 노이즈가 발생하지 않나요?
  - 핵심 지표만 선별하는 기준은?

**고려사항**:
- 모니터링 인프라
- 알림 피로도
- 운영팀 역량
- 비즈니스 임팩트

---

### Q6-2. 데드락 발생 시 대응 전략은?

**상황**:
- 데드락은 예방이 최우선이지만, 발생 시 대응 전략 필요
- 자동 해제 vs 수동 개입

**질문**:
- 실제 서비스에서 데드락 발생 빈도는?
- 데드락 발생 시 어떻게 대응하나요?
  - DB 자동 해제?
  - 수동 개입?
  - 알림 후 조사?
- 데드락 발생을 "정상"으로 볼 수 있나요?
  - 예: 매우 드물게 발생 (월 1회)
  - 예: 자동 해제되어 사용자 영향 없음
- 데드락 발생 시 어떤 조치를 취하나요?
  - 코드 수정?
  - 락 순서 변경?
  - 트랜잭션 범위 조정?

**고려사항**:
- 발생 빈도
- 사용자 영향
- 대응 비용
- 근본 원인

---

## 🎯 카테고리 7: 비즈니스 모델별 차이

### Q7-1. 이커머스 vs 금융 vs 소셜 서비스의 차이는?

**상황**:
- 서비스 특성에 따라 정합성 요구사항이 다를 수 있음
- 이커머스: 재고 정확성 중요
- 금융: 모든 거래 정확성 필수
- 소셜: 좋아요 수 약간의 오차 허용

**질문**:
- 실제로 서비스 특성에 따라 설계가 달라지나요?
- 같은 기능(예: 포인트 차감)이라도 서비스마다 다른가요?
  - 이커머스 포인트: 약간의 오차 허용?
  - 금융 포인트: 절대 오차 불가?
- 서비스 특성 외에 다른 요인이 있나요?
  - 규제 요구사항?
  - 고객 기대치?
  - 경쟁 서비스 수준?

**고려사항**:
- 서비스 특성
- 규제 요구사항
- 고객 기대치
- 경쟁 환경

---

### Q7-2. 스타트업 vs 대기업의 차이는?

**상황**:
- 스타트업: 빠른 개발, 약간의 리스크 허용
- 대기업: 안정성 최우선, 리스크 최소화

**질문**:
- 조직 규모에 따라 설계 결정이 달라지나요?
  - 스타트업: Optimistic Lock 선호? (빠른 개발)
  - 대기업: Pessimistic Lock 선호? (안정성)
- 실제로 어떤 차이가 있나요?
- 조직 규모가 커질수록 어떤 방향으로 변화하나요?

**고려사항**:
- 조직 문화
- 리스크 허용도
- 운영 역량
- 비즈니스 우선순위

---

## 🎯 카테고리 8: 실무 경험과 베스트 프랙티스

### Q8-1. 실제 서비스에서 가장 많이 발생하는 문제는?

**상황**:

**질문**:
- 실제 서비스에서 가장 많이 발생하는 transactional operation 문제는?
  - 데드락?
  - 락 경쟁?
  - 데이터 불일치?
  - 성능 저하?
- 어떤 도메인에서 문제가 가장 많이 발생하나요?
- 문제 발생 시 가장 효과적인 해결 방법은?

**고려사항**:
- 실제 경험
- 문제 빈도
- 해결 난이도
- 비즈니스 임팩트

---

### Q8-2. "과도한 최적화" vs "충분한 최적화"의 기준은?

**상황**:
- 이론적으로는 많은 최적화가 가능하지만, 실제로는 "충분한" 수준에서 멈춤

**질문**:
- 실제 서비스에서 "충분한 최적화"의 기준은?
  - 성능 목표 달성?
  - 비즈니스 요구사항 충족?
  - 운영팀 수용 가능 수준?
- "과도한 최적화"를 피하는 기준은?
  - 복잡도 증가?
  - 유지보수 비용?
  - ROI?
- 언제 추가 최적화를 중단하나요?

**고려사항**:
- 비즈니스 가치
- 기술적 복잡도
- 유지보수 비용
- 팀 역량

### Q8-3. 실무에서 가장 중요한 원칙은?

**질문**:
- 실무에서 가장 중요한 transactional operation 원칙은?
  - 정합성 보장?
  - 성능 최적화?
  - 운영 편의성?
  - 코드 가독성?
- 원칙 간 충돌 시 어떻게 우선순위를 정하나요?
- 신입 개발자에게 가장 먼저 알려줄 원칙은?

**고려사항**:
- 실무 경험
- 조직 문화
- 비즈니스 우선순위
- 팀 역량
